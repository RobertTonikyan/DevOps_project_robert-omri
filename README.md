## 🏗️ ארכיטקטורת המערכת

המערכת הונדסה ונבנתה בצורה מאובטחת ומבוזרת, המבוססת על ארכיטקטורת שלוש שכבות (3-Tier Architecture) ומנוהלת במלואה באמצעות קונטיינרים וכלי הניהול Docker Compose.

### 1. שכבת מסד הנתונים (Database Layer)
* **הטכנולוגיה שנבחרה:** **PostgreSQL (גרסת 17-Alpine)**.
* **נימוק:** הבחירה ב-PostgreSQL נעשתה בשל היותו מסד נתונים יחסי (Relational) חזק, יציב, בעל מנגנוני אבטחת מידע קשיחים ותמיכה מלאה בתקני ACID המחמירים. השימוש באימג' מבוסס הפצת **Alpine Linux** נבחר במכוון על מנת לצמצם את נפח הקונטיינר למינימום האפשרי, ובכך **להקטין בצורה דרסטית את שטח התקיפה** (Attack Surface) מפני חולשות אבטחה ופרצות פוטנציאליות ברמת מערכת ההפעלה (CVEs).

### 2. הפרדת רשתות (Network Segmentation)
כדי לממש את עקרון זכויות היתר המינימליות (Principle of Least Privilege) ולמנוע גישה ישירה לרכיבי התשתית הרגישים, המערכת מחולקת לשתי רשתות וירטואליות מבודדות (`Bridge Drivers`):
* **`frontend-net`:** מאפשרת תעבורת רשת אך ורק בין שרת ה-Reverse Proxy (Nginx) לבין שרת האפליקציה (Node.js).
* **`backend-net`:** מגבילה ומאפשרת תקשורת אך ורק בין שרת האפליקציה (Node.js), מסד הנתונים (Postgres), וממשק הניהול הגרפי (pgAdmin).

> 🔒 **משמעות אבטחתית:** בסיס הנתונים מבודד לחלוטין ומנותק לחלוטין מגישה ישירה של פורטים חיצוניים לעולם החיצון. כל ניסיון סריקה או חדירה מחוץ לפרימטר ייכשל באופן מוחלט, שכן תעבורת הנתונים חייבת לעבור סינון, ולידציה ואימות דרך לוגיקת האפליקציה בלבד.

### 3. שרת ה-Reverse Proxy ובקרת גישה (Nginx)
קונטיינר ה-Nginx משמש כשער הכניסה היחיד והמאובטח למערכת (Perimeter Gateway) ומאזין לפורט החיצוני `8080`. מעבר לניתוב וחלוקת העומסים לאפליקציה, הוגדרו רשימות בקרת גישה קשיחות (ACLs) ישירות בתוך קובץ הקונפיגורציה של השרת (`default.conf`).
* כל תעבורה נכנסת המנסה לגשת ישירות לנתיב הניהול הרגיש `/admin-panel` נחסמת באופן מיידי ברמת ה-Proxy, ומחזירה שגיאת **`403 Forbidden`**. הגנה זו מונעת תקורת חישוב מיותרת מהאפליקציה וחוסמת ניסיונות ניצול לרעה (Exploitation) של קוד השרת עוד לפני שהוא מגיע לשכבת ה-Node.js.
