## 🇮🇱 עברית - הסבר ארכיטקטורה והוראות

### 🏗️ ארכיטקטורת המערכת

המערכת בנויה בצורה מאובטחת, מבוזרת ומבוססת על ארכיטקטורת שלוש שכבות (3-Tier Architecture) המנוהלת במלואה באמצעות Docker Compose.

#### 1. בסיס הנתונים (Database Layer)
* **הטכנולוגיה שנבחרה:** **PostgreSQL (גרסת 17-Alpine)**.
* **נימוק:** בחרנו ב-PostgreSQL מכיוון שהוא מסד נתונים יחסי (Relational) חזק, יציב, קשיח מבחינת אבטחת מידע ותומך ב-ACID בצורה מלאה. השימוש באימג' מבוסס **Alpine Linux** נבחר במכוון כדי להקטין את גודל הקונטיינר למינימום ובכך **לצמצם את שטח התקיפה** (Attack Surface) מפני חולשות אבטחה (CVEs).

#### 2. הפרדת רשתות (Network Segmentation)
כדי למנוע גישה ישירה לרכיבים רגישים, המערכת מחולקת לשתי רשתות מבודדות (`Bridge Drivers`):
* **`frontend-net`:** מחברת אך ורק בין ה-Reverse Proxy (Nginx) לבין שרת האפליקציה (Node.js).
* **`backend-net`:** מחברת אך ורק בין שרת האפליקציה (Node.js), בסיס הנתונים (Postgres), וממשק הניהול (pgAdmin).

> 🔒 **משמעות אבטחתית:** בסיס הנתונים מבודד לחלוטין מהעולם החיצון. פורץ שינסה לסרוק פורטים מבחוץ לא יוכל להגיע או לדבר ישירות עם ה-Database, אלא יהיה חייב לעבור דרך שכבות ההגנה והאפליקציה.

#### 3. ה-Reverse Proxy והגנת נתיבים רגישים (Nginx)
שרת ה-Nginx משמש כנקודת הכניסה היחידה למערכת (בפורט `8080`). מעבר לניתוב הבקשות לאפליקציה, הגדרנו בתוך קובץ ה-`default.conf` שלו חסימה קשיחה (Access Control) עבור נתיבים רגישים.
* כל ניסיון גישה לנתיב הניהול הרגיש `/admin-panel` נחסם ישירות ברמת ה-Proxy ומחזיר שגיאת **`403 Forbidden`** עוד לפני שהבקשה מגיעה בכלל לשרת ה-Node.js.
